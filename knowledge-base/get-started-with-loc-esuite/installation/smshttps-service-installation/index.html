SMSHTTPS Service – Installation

PostedOctober 14, 2021
UpdatedOctober 14, 2021
Bydava-LOC

Information

The SMSHttps Service is the communication core of LOC SMS V4.5.0 and a merger of L_SMS, APIHttps and L_WIN (optional) applications.

It is strongly recommended to use SMSInstall to access the Company setup section of Launchpad since the SMSHttps service must be stopped to install/upgrade options and restarted once the operation is completed, SMSInstall does it automatically.

To consult the Company Setup settings (i.e. read-only), click on SMSStart – Configuration – Update Tab, Company Setup button. This does not stop the service. However, do not change any settings or install options and samples (go through SMSInstall as outlined above).

Once the SMSHttps service is installed, follow the instructions in this document, Hybrid SMS programs to adjust SMSStart accordingly.

NOTE: For hardware communication, see Hardware_setup.
During an update of options when SMSHttps service is already installed, SMShttps service will be set by SMSinstall.exe for all service failure actions to restart after 120 min. After the update is completed, all service failure actions will be set to restart after 1 min.

Prerequisites and dependencies

The prerequisites are: SQL Server 2012 SP2 or more recent, Microsoft .NET 4.6.2 or higher, Powershell 5.0 and  VCredist  2015.  All these prequistes  can be installed with SMSinstall. The SQL Server engine collation and the Storeman database (STORESQL/LANESQL) collation must now absolutly match.

  • To find out the SQL server engine collation, execute this query: SELECT CONVERT (varchar(256), SERVERPROPERTY(‘collation’));
  • To find out the databases collation, execute this query: SELECT name, collation_name FROM sys.databases

A guide to install an SQL Server engine with the same collation of a Storeman database can be found in SMSMembers.chm under System/Database/Manual installation of SQL Server.

Windows 8.1 or Windows Server 2012 R2

WARNING: There is a known issue when the software is installed on Windows 8.1 or Windows Server 2012 R2, we have a solution here: Issue With Installs on Windows 8.1 or Windows Server 2012 R2

Software Programming key

No element required on the software key.

Installation

To install the SMSHttps service, follow these steps:

  1. Open SMSInstall utility from the Storeman folder
  2. Click SMS V4
  3. Click on Install Services to open the SMS v4 installation window
  4. From the Install SMS V4 window, add the check mark on the right of the services you want to install:
    •  Https svc (64-bit): for 64-bit operating systems
    •  Https svc (32-bit) for 32-bit operating systems, note that it can be installed on 64-bit operating systems

If the SMSHttpsService service is to be installed indepentely of the Storeman installation, the note on top of the terminal type will read “New v4 installation“

Configuration Tips

The configuration will  be set in the SMSHttps.ini file located into the Storeman\SMSHttps64\ folder for the 64-bit folder.   

To list the available ports that can be used in the [HTTPSERVER] and [WINSERVER] section, in a Command Prompt window, type: netstat -ano. This will list the occupied ports in the Local adress column.

The PID number can be used to fiind what is the application using a port. This can be done using task Manager — details tab. View all the local adresses definition and port used to make sure the port that will be used in smshttps.ini are not already used. For example, let us look for the local adress 0.0.0.0 port 1433 that is used by the PID 11216.

It is found that PID 11216 is associated to sqlservr.exe. This means that the SQL Server engine is listening to port 1433. 

It is also possible that the port numbers that will be used in SmsHttps.ini need to be authorized in Windows Defender Firewall or any other network monitoring tool that can be in place. Verify with your IT network manager if needed.

SmsHttps.ini Parameters

Authentication

According to PCI DSS, user, password and passphrases must meet these requirements:

  • Require a minimum length of at least seven characters
  • Contain both numeric and alphabetic characters.
  • Change user passwords/passphrases at least once every 90 days
  • Do not allow an individual to submit a new password/passphrase that is the same as any of the last four passwords/passphrases he or she has used
  • Using strong cryptography, render all authentication credentials (such as passwords/phrases) unreadable during transmission and storage on all system components

Active Directory/Windows Authentication

To login using Windows Authentication (Active Directory or Local if the computer is not joined to a domain), use the following configuration:

Applicable to both [HTTPSERVER] and [WINSERVER] server:

  • AuthMeth=AD
  • ADDomain=XXXXXXXYYYYYZZZZZ where XXXXXXXYYYYYZZZZZ is the Domain name. If the computer is not joined to a domain, set it to “localhost”.

Make sure there is an operator with the “Email” field that match [Username]@[ADDomain value] where [Username] is the user name and ADDomain is the domain name. If the computer is not joined to a domain, you need to use “localhost”.

Note that you must append “?authmeth=AD” in the browser’s URL (ie https://localhost/esuite/sys/index.html?authmeth=AD)

Azure Active Directory

To login using Azure Active Directory, use the following configuration:

  • AuthMeth=Azure
  • OAuth2TenantID=XXXXXXXYYYYYZZZZZ
  • OAuth2ClientID=XXXXXXXYYYYYZZZZZ

Make sure there is an operator with the “Email” (must match user’s email address) and “Short name” field must be populated.

Note that you must append “?authmeth=Azure” in the browser’s URL (ie https://localhost/esuite/eSuite_Care/index.html?authmeth=Azure)Setup WINSERVER TokenPassphrase for the use of “Uplink” and “Exchange files over Https (apiXchFTP.dll)”

  • (Optional) AuthMeth=NONE
  • TokenPassphrase=XXXXXYYYYYYZZZZZZ
  • (Optional) TokenExpire=XXX
  • (Optional) TokenOrigin=xxx.ggg.ca:port
  • These keys (TokenPassphrase,TokenExpire) are applicable only for WINSERVER. You must add TokenPassphrase in the Server side and in the Client side using SMSCertificate.exe. The passphrase used to create this key should be the same in both side. On the client side, the [WINSERVER] may not be running. AuthMeth=NONE in [WINSERVER] is used to reject all the http requests that have \scripts\trs.exe in the url_path and they were sent to the service without Authorization header. In other words, it is to not allow the client to request eStore, eSuite, mStore using the Webbrowser through the [WINSERVER].

How the SMS-Token expiration time is validated

  • When the client is going to make an exchange with the server he makes a request “GET” or “POST” and the client must send a Token in the http header that have an expiration time. The tokens are different always in each request, not only because of the exp claim but for other claims that are calculated at the time of placing the request.
    The expiration time is the same time as the request time. Then if exp+TokenExpire ‘>’ Now the token is valid, if exp+TokenExpire ‘<‘ Now it means that the Token is expired and is invalid. The difference between exp+TokenExpire+network latency time and Now have to be always a positive value; if the result is negative the token is invalid.
    Inside of the software this time is expressed in Unix epoch, which is the number of seconds that have elapsed since January 1, 1970 (midnight UTC/GMT), not counting leap seconds. That is why the validation is performed perfectly well even if the server and the client are in different times zones.

Start the service

To start the SMSHttps service, follow these steps:

  1. From the Storeman folder, launch the L_WIN once and close it right away to set the computer name and other configurations as well
  2. Open SMSStart, Click on Configuration, then from the Tasks tab unselect “Monitor health” and “Execute at startup” for Launchpad and SMS
  3. Open the Windows Service manager (access can be from either Task Manager or Control Panel)
  4. Find SmsHttps_Storeman in the list and open its properties
  5. Under the General tab, set the Startup type to Automatic
  6. Under the Log On tab, select This account and set the Windows login account and Password that runs the service.
    • If you want to use the auto-update feature, the user MUST be a local administrator AND a member of either the SMSProgrammers or SMSOperators group
    • If you do not want to use the auto-update feature, the user MUST be a member of SMSProgrammers or SMSOperators group
    • In ALL cases, the user MUST also exist on the lower levels servers/lanes for the deployment to work.
  7. Under the Recovery tab, make sure that the First failure, Second failure and Subsequent failures are set to Restart the Service.
  8. Select SmsHttps_Storeman in the list of services  and start the service
  9. Logs can be checked to confirm that no errors occurred during the startup of the service. They2 are located in the \Storeman\log\ folder

Monitoring SMSHttps Service after it is started

To test if SMSHttps service is listening to a port define in the sections [HTTPSERVER] and [SMSSERVER], type in the address bar of web broswer running on the pc where SMSHttps Service is running: localhost:(port number). To test is from another computer , type in the address bar the IP and port of the computer where the service is installed. Example: 192.168.0.20:8000.

The example shown test if SMSHttps service is listening to the port 8000. The answer received is the version and build date of the SMShttps service running in this machine

These logs (in \logs\ folder) are created or used by SMSHttps service and can be used to verify the state of the service at startup: *_SMSHttps.sls, *_WinHttps.sls, *_WIN_*.sls, *_TRS_*.sls.

Troubleshooting

There may be instances when the service fails to start. In most cases, this is caused by one of the following reasons:

NOTE: The following points should be checked first. If the problem persists after these checks, contact LOC Software.

  1. The port defined in http://SeverLocalIP:port is already in use by another application or service

    Replace the port with another unused port, typically greater than 1059. If port 80 is already in use, try 8080, 8888 or 81
  2. The user defined on the service does not have access to the database

    The user set must be part of either SMSProgrammers group or SMSOperators group. Also check directly in SQL to see if the user or groups have access to the database
  3. L_WIN.exe was not started initially, and then closed to set the computer name. Start the launchpad then close it and then start the SMSHttps service
