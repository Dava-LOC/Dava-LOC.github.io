SMSInstall.ini, Update.ini and Version.ini files

PostedOctober 14, 2021
UpdatedOctober 14, 2021
Bydava-LOC

Information

This document provides a place where issues regarding the SMSInstall.ini, Update.ini and Version.ini files can be discussed. The goal is to provide a unique location to contain the documentation on all aspects of these configuration files. It does not serve to replace documentation found elsewhere, but it acts as a complement to it.

SMSInstall.ini, Update.ini and Version.ini roles

Originally, only one INI file was used to contain configuration information for the installer utility. As we added dynamic values to be stored in this file, it became evident that performing updates could clash. It became necessary to split the original SMSInstall.ini file to protect the local data that must not be changed from any update activity that would install or download over the installer configuration file. So, the Update.ini file is created to contain sections dynamically created and maintained. SMSInstall.ini would keep the original parameters intended to be downloadable. Such downloadable sections serve to configure the installer behavior and default values.

These INI files are normally not provided on our FTP server. If any third party wants to have a customized installer, he has to provide his own file. The Update.ini file will now contain all the dynamic data that was locally assigned. This file is not supposed to be present on our FTP. Even if accidentally present, our downloader will not pick it up during the download process. This way, this file should not get accidentally erased. The operation of copying the installation files during installation on the server also skips this file if accidentally present.

There is however a direct interrelation between the SMSInstall.ini file and the Update.ini file when the SMSInstall utility performs a new installation. Several sections normally found in the Update.ini file can be put into the SMSInstall.ini file to preset them for customizing prior to installation. Those sections are the SAMPLES, FTPSYNC, ALTFTPSYNC and SHARENAMES. By supplying the correct definitions into these sections, it is possible to set their value in the Update.ini file by supplying the sections in the SMSInstall.ini file. By example, if a sample is defined in the SAMPLES section for a workstation, when this workstation will be first installed, the sample will be selected already when it is time to install the samples on that workstation.

The Version.ini file is normally present on the FTP server. It identifies the version currently present on the server. If the file is absent, it is an indication that the FTP server is in maintenance and that downloads are not permitted.

FTP server issues

The SMSInstall.ini file is not to be provided on the LOC Software FTP server. A third party can mount his own FTP server and place his own SMSInstall.ini file on it. The download operation can pickup the SMSInstall.ini file and place it in the local install folder. This file can be used to change the defaults values of the installer and change its behavior. This way, third party can create a customized installer that will propose their own choice for architecture.

The SMSInstall.ini file

This file contains configuration parameters specific to create a customized installer. It controls some behavior of the installer and provide default values for the different types of installations. It contains the custom, common, server, thin and fat sections. This file may be provided by a dealer that wants to have a customized installation.

Pre-configuring an installation

It is sometimes desirable to preset the installation defaults to a different value than the defaults of the SMSInstall utility. A good example of this is when selecting to install by default to a different drive than the C:\ or to a different folder name than STOREMAN. To streamline the installation, it is highly preferable to preset those parameters so that the technician installing the system has nothing to do other than push the next button at each step.

There is a simple way to preset many of those parameters easily. What you need to do is to have the installer files locally in a drive where you are allowed to write files to. You cannot perform this by running the software from an installation CD. What you need to do is to prepare to install your SMS Software and answer or set all the parameters that you desire. Then, when you reach the point where you would launch the SMS installation by pushing the Proceed button, there is another button available on that screen saying [Save Configuration]. When you push this button, all the selected parameters will be saved into the SMSInstall.ini file where the SMSInstall.exe is run from.

You can also do the same for installing front end workstations and backoffice workstations. Different sections will be created containing the gathered installation configuration. Once your installation is complete, the SMSInstall.ini file would contain all the selected configuration values of your current installation. You can then send this file along with the installation files so that new installations will automatically have the same default values as the ones you have selected.

[CUSTOM] section

This section contains keys to customize the behavior of the SMSInstall utility. It was created following a request by a chain of stores that wanted to control some aspects of the installation process.

SECURITY key

The SECURITY key is used to prevent the installer program from installing the user security. When this key is set to 0, the user security module will not be presented in the list of modules to install. If the key is set to 1 or if the key is not present, the user security module will be present in the list of modules to install. It is expected that some form of user security will be implemented by the administrator installing the system.

SHARE key

The SHARE key is used to prevent the installer program from installing the sharing of files and resources. When this key is set to 0, the share module will not be presented in the list of modules to install. If the key is set to 1 or if the key is not present, the share module will be present in the list of modules to install. It is expected that the required resources to operate the SMS software will be appropriately shared by the administrator installing the system. For a list of resources that require sharing, refer to the SMSInstall share files technical document.

COPYINSTALL key

The COPYINSTALL key is used to prevent the installer program from copying the installation folder to the target server installation path. When this key is set to 0, the copy installation files module will not be presented in the list of modules to install. If the key is set to 1 or if the key is not present, the copy installation files will be present in the list of modules to install. Note that several functionalities are expecting to find this folder on the server at this location but a separate location can be used.

[COMMON] section

This section contains keys to customize the default values used by the SMSInstall utility. The common section contains configuration common to all types of software installations.

InstallPath key

This key was created to allow customization of the installer default installation path. This key is used on new systems as the default location to install the programs. The data path and other items are dependent on this default and will follow under the same path.

The DataPath key

This key defines the data path for server and front end workstation. It is recommended not to use unusual values for this key. It should be left to the default or undefined. Having a nonstandard data path is not completely supported by the SMS software and a few tweaks are required to implement it.

Store key

This key contains the store number. This value is valid for all workstations of the current instance. It is used by the installer when installing new workstations as the default store value to use.

The share names keys

There are two network share names that can be set. The first one is used to point to the SMS instance main folder. The second share name is usually not used and would point to a separate data path if one has been configured. It is not recommended to use a path for the data files other than the standard path. The keys are ProgramShareName and DataShareName. The network share names are used to define the shares that will be put in place on each sms instance. For uniformity, it is recommended to have a common name be used for all the shares.

The User Security parameter keys

This set of keys define the main configuration for user security that is common to all workstations. They comprise of the keys OperatorUser and RunAsUser which are set as 1 or 0 for a true or false values. The selected operator name is saved in the OperatorName key. There are also two encrypted keys for the corresponding password definitions for the operator and runasuser users: OperatorPassword and RunAsPassword. There are additional keys for configuring the user security but those are defined in the Server, Fat and Thin sections since their configuration can differ from each other.

Server connection parameter keys

This set of keys define the connection parameters to use for any workstation to connect to the server folders. It also defines if the server has dual architecture for proper mail slot connection configuration. Once set for one station, these parameters should be valid for any workstation. They comprise of the keys ServerName, ServerProgramPath, ServerDataPath, ServerMailSlot and DualServer. The dualserver key is a true or false key using 0 or 1 as value.

Server database connection parameter keys

This set of keys in the COMMON section define the server database connection parameters. Those parameters are common to all workstations that will connect to the server. They comprise of the keys ServerSQLServerName, ServerSQLUserName, ServerSQLPassword, ServerDBName and ServerUseWinAuth. The ServerUseWinAuth key is a true false type key using 0 or 1 as data value. Those keys are used by the front end and back-office workstations.

[SERVER] section

The SERVER section comprise of keys that store the selected server configuration. When installing a server, if this configuration exist, it is loaded instead of using the embedded defaults. These keys comprise the following: DualServer, LnkProgram, LnkDesktop, LnkStartup. The DualServer key defines if the server is using dual Launchpad applications. As for the Lnk keys, they define where the shortcuts will be present.

This section contains also keys that define the local database connection parameters. They are SQLServerName, SQLUserName, SQLPassword and DBName. There is also the UseWinAuth key which is a true false key with values of 0 or 1. The user and password keys are required for SQL authentication only. Using only windows authentication is recommended.

There are also three keys that refer to user security installation options. They are OperatorAutoLogin, RunAsShortcutDesktop and RunAsShortcutPrograms Those keys are true false type using values 0 or 1. They control the options that can customized per workstation model.

There is a key used mainly for distributed host systems. The key name is LocalPath. In distributed systems, there are two or more processors handling the charge load of the host. In this configuration, a file server is used to hold the storeman folder and the processors access this folder through the network. Part of the software installation has to be local to the processor and this LocalPath key is the placeholder for preconfiguring the local path where the SMSStart utility should be copied to. Note that installing a distributed host involves manual configuring and cannot be automated.

[THIN] section

The THIN section comprise of keys that store the selected thin configuration. When installing a THIN workstation, if this section exist, it is loaded instead of using the embedded THIN defaults. This section contains the following keys: LnkProgram, LnkDesktop, LnkStartup. These three keys define which shortcuts will be installed.

There are also three keys defining the user security installation options. They are OperatorAutoLogin, RunAsShortcutDesktop and RunAsShortcutPrograms. Those keys are of type true false using 0 or 1 as data value.

[FAT] section

The FAT section comprise of keys that store the selected FAT configuration. When installing a FAT workstation, if this section exist, it is loaded instead of using the embedded FAT defaults. This section contains the following keys: LnkProgram, LnkDesktop, LnkStartup. These three keys define which shortcuts will be installed.

The section also contains keys for defining the SQL connection parameters. Those keys are SQLServerName, SQLUserName, SQLPassword, DBName and UseWinAuth. The UseWinAuth key is of type true false using values 0 and 1. It is recommended to use Windows authentication for security reasons.

Additionally, the ServerMailSlot key in this section serves to define the connection to the server mail slot.

There are also three keys defining the user security installation options. They are OperatorAutoLogin, RunAsShortcutDesktop and RunAsShortcutPrograms. Those keys are of type true false using 0 or 1 as data value.

There is a key used mainly for distributed systems. The key name is LocalPath. In this configuration, a file server is used to hold the storeman folder and the processor access this folder through the network. Part of the software installation has to be local to the processor and this LocalPath key is the placeholder for preconfiguring the local path where the SMSStart utility should be copied to. Some manual intervention is required to complete the installation.

[SAMPLES] section

The SAMPLES section in the SMSInstall.ini file serves only one purpose: it is used for presetting the samples list for a new installation. This section will be transferred to the Update.ini file during the installation of programs and then, will serve as predefined values for samples to be installed. Note that this section used to be in the SMSInstall.ini file in older versions of SMS before being moved to the Update.ini file so that it can be locally customized.

For the reference sample to work fine, it must contain the Samples.ini file which serves as the list of installed options and samples. Said samples should be present in the installation folder in the SAMPLES folder. It is important to understand that the Samples.ini file contains only the list of installed samples. If the reference sample was not installed, the Samples.ini file will not contain it.

For the preparation cycle to be complete, you must first create the reference sample by looking for custom files and adding all customs into the reference sample. At this point, the samples.ini file does not yet contain the reference sample entry because creating the reference sample did not install it. Once the sample is created, you must install it so the entry appears in the samples.ini file. Installing it will modify the samples.ini file by adding this reference sample as an installed sample. Then, you check for custom files again and the Samples.ini file will again appear as a modified file. Adding this file to the reference sample will add the reference sample installed entry. At that point, listing the samples to be installed from the reference sample in the samples tab will include the reference sample.

[FTPSYNC] section

The FTPSYNC section in the SMSInstall.ini file serves only one purpose: It is used for predefining the FTP server to use for downloads. This section will be transferred to the Update.ini file during the installation of programs only if the keys it contains do not exist within the Update.ini file. Note that this section used to be in the SMSInstall.ini file before being moved to the Update.ini file so that it can be locally customized.

[ALTFTPSYNC] section

The ALTFTPSYNC section in the SMSInstall.ini file serves only one purpose: It is used for predefining the alternate FTP server to use for downloads. This section will be transferred to the Update.ini file during the installation of programs only if the keys it contains do not exist within the Update.ini file. Note that this section used to be in the SMSInstall.ini file before being moved to the Update.ini file so that it can be locally customized.

Update.ini file

This file contains configuration parameters specific to administer the download and updates. It also serves as a repository for other managing tasks the installer or the SMSStart utility may implement. In general, this file should not be modified by the user directly.

[UPDATE] section

The UPDATE section is used to manage automatic updates. This section is seen by all workstations attached to this server. All workstations are supposedly installed using the files from the current install folder where this file resides and therefore all workstation will monitor this Update.ini file for status of updates.

Version key

This key contains the version tag of the last successful download operation. This key may not be set yet if no download was ever performed. The content of this key is compared with the saved value in the registry of each station for equality. If the values are different, it means that the installed versions are different than the available version in the install folder and an update is therefore available. When the update is run, the content of this key is copied to the registry of the workstation being updated thus saving the last update version tag.

Workstation key

This key is a flag set by the server to indicate to the workstations that belong to this group are allowed or not to perform the update. The point is that the server is to be updated first and when this operation is completed, only then the workstations can be updated. This key takes only values 0 or 1. The download operation sets this flag to 0 to prevent the workstations from updating. When the server updates itself, it sets the flag to 1 to unlock the workstations updates.

METHOD key

The METHOD key contains the schedule method to implement for auto-updates. This key can take the following values: TIMEZONE, MAINTENANCEZONE, IDLE, ASAP and NONE. These values represent the scheduling logic to consider for running the auto-updates. The ASAP means to run the updates as soon as possible. So updates could be run even when someone is logged in but no activity is recorded for a while. the IDLE state is when the user is not logged in. Both zones imply that we want ASAP but only during a specific time zone. If TIMEZONE is selected, the defined time zone applies. If MAINTENANCEZONE is selected, then the maintenance time zone applies. This time zone can be configured in the SMSStart utility.

STARTTIME and ENDTIME keys

The STARTTIME and ENDTIME contain the beginning and ending time for the TIMEZONE method. If the ENDTIME is smaller than the STARTTIME, roll over midnight is assumed and the defined period extends to the morning.

LOCKTHIN.TXT file key

This file acts as a stub file named LOCKTHIN.TXT. When this stub file is present into the \Storeman\Install\ folder, the workstations are locked.

DisableWorkstationLock key

This key is a hidden key. In most systems, it should not be present. When this key is set to the value “1”, Workstation locking and automatic SMS updates are disabled. This key is designed to be used on distributed host systems where several processors are running multiple launchpads to process the data from a big chain of stores. Updating such a system while it is running is not possible. To proceed with an update, the system should be first shut down and rebooted to insure that all resources are properly freed and that the update can proceed. Such a procedure would be performed very intermittently to minimise downtime impact on the chain.

SQL prerequisite selection keys

There is a key for each SQL prerequisite model. Those keys exist to remember the model selection made by the client. They are used for the automatic download of updates if the prerequisites as a whole are allowed for download in the L_PRO application configuration section. The keys are SQL2012X86, SQL2012X64, SQL2014X86, SQL2014X64 and SQL2016X64. They bear a true or false value represented by 1 or 0.

There is an additional set of keys that define the languages selected for the prerequisites. There are two keys used to record the user selection of languages. They are PREREQ_EN and PREREQ_ES for English and Spanish respectively. For any download to be selected, both the language and the individual package type must be selected.

Shortcuts selection keys

There is a set of three keys in the UPDATE section that serve to remember the selection of shortcuts to install. Those selections are then used as the default for the next time the user updates the system manually. There is only one key for the server, the lane and the thin workstations. The value assigned to those keys consist of a bit map where the lower to higher bits are assigned to program, desktop and startup shortcuts. So selecting all shortcuts would take value 7 while selecting none would take value 0. Therefore, the configuration is remembered by type of workstation. The names of the keys are ServerShortcuts, LaneShortcuts and ThinShortcuts.

DOWNLOAD_ALL_OPTIONS key

The DOWNLOAD_ALL_OPTIONS key is not tied to any configuration screen. It is not controlled either by the program. It is used to tell the SMSInstall utility to download the options as a whole like it used to do it in the older versions. In the new versions, the user selects the options he needs for his store. However, you may want to get all available options all the time like it used to be done. Setting this key to 1 will activate it. When set, the options selector is no longer presented and the user has no choice but to download all options.

AllOptionsSelected key

The AllOptionsSelected key is not tied to any configuration screen. It is controlled by the options selector. Its purpose is to remember that all options were selected using the select all options button in the options selector. Its purpose is to force download any new option that is added to the version automatically. In the new versions, the user selects the options he needs for his store. However, when the user selects all options, he may expect that new options will automatically be downloaded as well. This is similar to the DOWNLOAD_ALL_OPTIONS key except that it leaves the options selector available to the user.

[SAMPLES] section

This section contains references to samples installed on the different workstations and server. It is used when reinstalling a dead workstation as an indication of which sample was installed as reference on specific workstations. The installer will use this information to reset the default sample value on the workstation being restored. The syntax of the key follows this rule: The name of the key starts with REFERENCE_ followed by the instance name. Next comes an underline as separator and then follows the store and terminal numbers. Here is an example of the complete name: REFERENCE_Storeman_125901. In this example, the instance name is Storeman; the store number is 125 and the terminal number is 901. The instance name is derived from the folder name in which the SMS software is installed (usually Storeman).

[FTPSYNC] section

This section contains the FTP connection parameters for the main FTP host for software downloads.

HOST key

The HOST key defines the FTP server name to use to download from. It can be a web name like FTP.LOCSOFTWARE.NET or an IP address or a machine name on your internal network.

FTPROOT key

The FTPROOT key defines the source folder at the FTP server that contains the version to download. Since a FTP host will host several versions or even different software, this parameter is required to designate the appropriate source for the update.

USERNAME and FTPPASSWORD keys

Those keys define the credentials to use to gain access to the FTP server for the download. If you setup your own FTP server, it is important not to broadcast to everyone a user name and password that would allow the holder to change the content of the FTP server. Instead, create a read only password access. Note that the FTPPASSWORD key is saved in an encrypted form to make it unreadable in the INI file.

FTP_CONNECT_TIMEOUT key

The FTP_CONNECT_TIMEOUT key defines the timeout value for the connection to the FTP host. This value should be long enough to cover for slow or busy servers and slow web connections.

PASSIVE_FTP key

The PASSIVE_FTP key defines a method used to access the data contained in the FTP server. This is a true or false value for either using or not using passive method. This parameter might need to be set to an appropriate value depending on the FTP server software version.

SSL key

The SSL key defines if SSL TLS security is to be used for accessing the FTP server and downloading data. This key can take either a true or false value. It is preferable to turn this option on to protect the content of the data being transmitted. The data will also be encrypted when this option is turned on. Understand that this option can only be turned on if your FTP server supports SSL protocols.

IMPLICIT key

The IMPLICIT key defines the manner into which the SSL connection will be established. By default, FTP transfers occur on port 21. If the implicit option is selected, the SSL connection will default to port 990 instead. Note that if you specify a port, it will always override the default port. Connections on port 990 assume that the SSL protocol will be negotiated thus the implicit character of this connection.

VALIDATE key

The VALIDATE key is a hidden key. It is not naturally present in the Update.ini file. It is used to disable the validation done on the root folder of the main FTP parameters when the FTP server selected is the LOC Software server. The validation is there to forbid jumping main version through the automatic update process. By definition, main version jumps should be run manually because there is always more to it than just an automatic update.

[ALTFTPSYNC] section

This section contains the FTP connection parameters for the alternate FTP host. The alternate host is only used to download samples. A dealer may set the main host connection parameters to the LOC FTP server and have his own FTP host for his customized samples. Only samples will be downloaded from this secondary host.

HOST key

The HOST key defines the FTP server name to use to download the samples from. It can be a web name like WWW.LOCSOFTWARE.NET or an IP address or a machine name on your internal network.

FTPROOT key

The FTPROOT key defines the source folder at the FTP server that contains files from the appropriate version to download. Since a FTP host will host several versions or even different software, this parameter is required to designate the appropriate source for the update.

USERNAME and FTPPASSWORD keys

Those keys define the credentials to use to gain access to the FTP server for the download. If you setup your own FTP server, it is important not to broadcast to everyone a user name and password that would allow the holder to change the content of the FTP server. Instead, create a read only password access. Note that the FTPPASSWORD key is saved in an encrypted form to make it unreadable in the INI file.

FTP_CONNECT_TIMEOUT key

The FTP_CONNECT_TIMEOUT key defines the timeout value for the connection to the FTP host. This value should be long enough to cover for slow or busy servers and slow web connections.

PASSIVE_FTP key

The PASSIVE_FTP key defines a method used to access the data contained in the FTP server. This is a true or false value for either using or not using passive method. This parameter might need to be set to an appropriate value depending on the FTP server software version.

SSL key

The SSL key defines if SSL TLS security is to be used for accessing the FTP server and downloading data. This key can take either a true or false value. It is preferable to turn this option on to protect the content of the data being transmitted. The data will also be encrypted when this option is turned on. Understand that this option can only be turned on if your FTP server supports SSL protocols.

IMPLICIT key

The IMPLICIT key defines the manner into which the SSL connection will be established. By default, FTP transfers occur on port 21. If the implicit option is selected, the SSL connection will default to port 990 instead. Note that if you specify a port, it will always override the default port. Connections on port 990 assume that the SSL protocol will be negotiated thus the implicit character of this connection.

SAMPLE_USE_ALT key

The SAMPLE_USE_ALT key is mandatory to enalble a connection to the alternate FTP  Itt also defines if the alternate FTP server is used to download the samples. This is a true or false value. If true, the alternate server connection is used. Note: the Version.ini file on both servers must contain the same version number. The dates can be different.

[SHARENAMES] section

The SHARENAMES section is used by the installer share module to store the network share name used when installing the share. In this manner, when it becomes necessary to reinstall the share module, the original share name will be reused automatically. Note that the share names are customizable only since version 3.3. If a separate data path is used, a second share name is required. This is why we have two possible types of keys in this section which are program and data keys. Each key also contains the terminal number in the key so that each terminal can have its own configuration.

PROGRAM_nnn key

Program keys contain the prefix PROGRAM_ followed by the terminal number as the full key name. The program key is a placeholder for the network share name used on that workstation for sharing the main installation folder. The default installation folder is STOREMAN and the default share name proposed is STOREMAN.

Prior to the version 3.3, the share name used was fixed to STOREMAN. This caused conflicts when installing more than one instance on one computer. The person installing had to manually change the network share names to separate the instances from each other. With version 3.3, the default share name used is the instance name which is the folder name in which the software is installed. There can not be two instances with the same instance name installed on one computer.

DATA_nnn key

Data keys contain the prefix DATA_ followed by the terminal number as the full key name. The data key is a placeholder for the network share name used on that workstation for sharing the office folder. The default share name proposed is the instance name followed by the suffix _DATA. It used to be fixed at STOREOFFICE.

This key is very rarely used. It is used only with customized systems where the data folder (office) is not located under the main installation folder. This architecture is not fully supported by the SMSInstall utility. Some manual intervention is required to properly configure such architecture. Unless there is a major strategic reason to select such an architecture, we do not recommended to use it.

Version.ini file

This file contains the identification of the version contained in the current folder. It also contains optional update behavior control. The file is used to validate downloads so that the correct version or the expected version is downloaded.

[UPDATE] section

This section is the only section present in the Version.ini file. It contains a mandatory key which is Version and an optional key which is Override.

Version key

The Version key contains the version identification for the version contained in the folder where the Version.ini file resides. This key can not be absent from the file. The format of the key must be maintained because specific parts are interpreted independently. The key contains two parts: the version number and the issue date. The date portion starts after the first space in the key value and is in the format YYYY/MM/DD. The version part is located at the beginning of the key value. It starts always with a V followed by four numbers separated by dots. Here is an example: V3.2.0.0 2009/04/03

Automatic updates will use the value found in the version.ini file and compare it to the currently installed version which is saved in the registry. If they do not match, then it is assumed that an update is available for installation. When downloading samples from a separate host, the first two numbers of the version part are compared with the alternate host version part and if they do not match, the download will not be allowed. This mechanism will protect against using mismatched versions of samples. In this test, the date part of the version tag is ignored.

Override key

This key is normally not present in the Version.ini file. It is used to direct special behavior mostly to the Launchpad application when executing automatic updates. When not present, the default value assigned to it is 1. This is a numerical value key. This value operates as a level. When set to a value, it encompasses all the values below the value set. This means that setting the value to 5 would include all the exceptions from 2 to 5. Here is the list of values:

  1. Default normal update
  2. Override custom detected
  3. Override mismatching version
  4. Override input required
  5. Override prerequisite missing
  6. Override automatic update not supported
  7. Override error

When a custom file is found and it is not integrated into the reference sample, if the override key is set to 1, the upgrade will not take place and an error will be reported. When the Override key is set to 2, instead of stopping on an error, the file is replaced with the updated file. This action is indicated in the normal mail of the update result. The conflicting files are listed in the mail messages.

When a custom file is found and it is integrated into the reference sample, installed or not, the update is performed but the file is not replaced, whether the Override value is 1 or 2. The result mail indicates that some files were not installed because they are part of other installed options. The clashing file is not listed but the clashing option is.

When there is an automatic update that jumps across a version boundary; like jumping from 3.2 to 3.3, This would cause an error unless the override level is set to 3.

Occasionally a human intervention is required to answer to a question about the current installation. In an automated update, nobody is there to answer and the process would abort on an error unless the override level 4 is set.

Some options require the presence of other options to be installed. These are called prerequisites. when updating an option that requires a prerequisite that is not present, the update would stop on an error unless the override level 5 is set.

Some older options use scripts or batch files that can not report an error when executed. Updating these options automatically is normally not allowed unless you set the override level to 6.

The normal behavior when an error occurs during the update is to stop the update and report the error. However, if it is desired that the update attempts to complete the update process anyway, the override level 7 is then required.

MainFTPControlsUpdates key

The MainFTPControlsUpdates key belongs only to the Version.ini file residing on the alternate FTP host. It does not apply to the Version.ini file found on the main FTP host for a good reason. This key is a flag taking either true or false values in the form of 0 or non zero (1). Its purpose is to relinquish control of the SMS updates to the version tag provided by the Version.ini file on the main FTP host. When an alternate FTP host is defined, the version tag in the Version.ini file on that alternate host becomes the controlling version tag for triggering the updates. Sometimes, it is preferable to let the main FTP host control the updates. A good reason to do this would be when the alternate FTP is only used for controlling Windows updates. In this case, the extra work to maintain the version tag in the Version.ini file of the alternate FTP host would serve no purpose. If the flag is not present, it is deemed taking the value of false (0).

The UPDATE_TYPE key

This key modifies the normal installation method for the options and samples. It controls the behavior of the automated installation. It can take one of the values 0, 1 or 2.
Value 0 stands for a normal install.
Value 1 will override all files.
Value 2 will install as first load.

The UPDATE_LANGUAGE key

This key serves to override the automatic detection of the language to install. It controls the behavior of the automated installation. This key can take only one of the following language codes:
EN for English
FR for French
ES for Spanish.
