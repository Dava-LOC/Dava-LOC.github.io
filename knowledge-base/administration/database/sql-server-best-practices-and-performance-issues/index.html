SQL Server Best Practices and Performance Issues

PostedOctober 21, 2021
UpdatedOctober 21, 2021
Bydava-LOC

Information

This document provides technical guidance about best practices for maintaining the LOC SMS databases on a SQL server instance. The goal of this document is to provide a list of summarized best practices.

Configuration at Head Office

In order to maximize SQL performance, it is desirable to configure TempDB with a reasonable minimum size and with reasonable auto growth settings. The intent is to avoid frequent automatic expansions of TempDB during processing, since each expansion would cause a request for allocation of Windows file space.
For large SMS environments, the TempDB database should be configured with a minimum size of 16 Gb, and with auto growth configured as 500 Mb. For very large SMS environments, the TempDB database should be configured with a minimum size of 32 Gb, and with auto growth configured as 1 Gb.

For similar reasons, the minimum size and auto growth for the SMS database at head office, should be set to reasonable values in order to avoid frequent auto growth. The auto growth settings for the SMS database for large and very large SMS environments should be configured similarly to the auto growth sizes as described above.

As per standard Microsoft recommendations for configuring SQL Server TempDB, multiple data files should be used at head office in order to improve concurrency.

SMS loyalty module

In general, for installations that use the SMS Loyalty module, it is important to minimize downtime of the Loyalty functionality at their stores, because customer dissatisfaction could occur in the event that a customer tries to use their accumulated points balance while the Loyalty functionality is not working. The result is that a SMS database that is used to track Loyalty information usually has a higher availability requirement than one that is not used to track Loyalty information.

Therefore, for such installations, it is important for management personnel to quantify what are their Recovery Time Objectives for the Loyalty functionality, such as the maximum tolerable downtime. Then, analysis of the objectives and analysis of the available hardware should be done to determine whether a gap in capability currently exists. If a gap exists, then further analysis should be done in order to determine what are the possible options to close the gap in capability, while also considering the option to accept the current situation (including documenting the gap and updating service level agreements (SLA) to reflect current capability).
In addition, disaster recovery planning needs to be done for the SMS database that contains the Loyalty tables, including planning for possible hardware failures.

Backup SMS Database

SMS databases should preferably be fully backed up on a daily basis, except for very large databases where a differential backup should be done daily along with at least one full weekly backup. The estimated duration of the full backup is the major deciding factor whether a full backup can be done daily. Databases should be backed up during a time period where the SQL Server is usually less busy, in order to minimize the duration of the backup.

For SQL RDBMS environments that use clustering, the databases on the secondary database server should also be backed up, in order to reduce risks of data loss in the event of certain recovery scenarios. For a more detailed explanation of best practices and options for backups and the performance trade-offs in a clustered environment, please refer to the Detailed Recommendations link (below).

For backups of the SMS database(s) at head office, it is recommended that backup rotation be used so that at least 2 weeks of local backup files are available to the DBA in case they are needed for a restore scenario or for a disaster recovery scenario. A copy of each backup file should also be retained offsite at a secure location or at a reputable cloud provider, and should be copied there each day as per standard best practices for disaster recovery planning. It may also be desirable to retain the backup files for a longer period at the offsite location in order to satisfy disaster recovery requirements and to satisfy regulatory requirements.

In order to meet the Recovery Time Objectives (RTO) that are initially proposed by management as part of disaster recovery planning, it is important that the data integrity of new backup files should be verified within 1 week by a combination of testing a restore of a live full backup file into a test environment and then running DBCC CheckDB (full, i.e. using the default with data_purity option rather than the physical_only option) on the restored database in that test environment.

Using backup compression in SQL Server can greatly decrease the duration of full backups but more CPU resources will be used while the backup runs. There is often a net benefit because there are often CPU resources available while SQL Server is waiting for the disk system where the full backup file is being written to. For SQL Server systems with 4 CPU cores or less, an analysis should be done by monitoring total CPU usage while a full backup is in progress. That can be used to determine whether using backup compression is a net benefit for the SQL environment.

For the databases at the store level, the SMS software database maintenance job retains, by default, 4 weeks of backup files on the local drives on each store server. It is now recommended to configure SMS to copy each backup file at the store level to a high availability network location or to a directory in the cloud.

Backup the system databases

The SQL System databases at head office (master, MSDB and model) should be backed up regularly, in case the SQL installation should ever need to be repaired later. The TempDB database does not need to be backed up, since it is maintained and re-built automatically by the SQL engine.

The SQL system resource file mssqlsystemresource.mdf (contains system metadata that is required by the SQL engine) cannot be backed up in the same way. After the initial installation of SQL Server at head office and also after upgrades to SQL Server, the file for the system resource database file mssqlsystemresource.mdf should be copied to a secure location, in case it is ever needed in the future for a downgrade / rollback / file corruption repair scenario.

At the store level, it would likely be acceptable to instead use a disaster recovery model of re-installing the local SQL Server (Express) and then restoring the databases from backup files that have been obtained, either from the local SMS store server or from head office, as needed, and then reprocessing SMS files on the store server as needed.

Backup transaction log at head office

For larger databases, such as the SMS database(s) at head office, the configured recovery model should be set to full rather than to simple. This would also require setting up a SQL Agent job to backup the transaction log every 15 minutes for each SMS database that is used at head office. This job will produce a transaction backup file with a unique filename at approximately 15 minutes intervals and the same job should also copy that transaction backup file to a secondary location (e.g. the same location where the full backup files are retained for the same database).

Retaining these transaction log backup files can be used as a method to reduce the risk of data loss and can sometimes reduce re-processing time in the event that the database needs to be restored from a backup. In recovery scenarios, retaining the transaction log backup files allows the option of restoring the database to a specific point in time after the point in time at which the database was last backed up (full or differential) by re-applying (using the restore command or the SQL Management Studio tool) any available transaction log backup files with a system date and time after the date and time of the last full or of the last differential backup file that was used during the restore. Such a point in time recovery could also possibly re-capture changes that were made during the day by users that were making changes using table maintenance functionality in the SMS application.

In a SQL Server environment that uses clustering or that uses log shipping, e.g. SQL 2014 Always On, the recovery model needs to be set to full due to the requirements of the SQL engine. In a clustered SQL Server environment, other procedural options, e.g. switching to the secondary database server, are also available when a database needs to be restored to a previous state, and these options should be planned and documented in advance as part of the disaster recover planning that is customized for each SMS installation at head office.

Backup at store level

The SMS databases at the store level are using SQL Server Express environments and they are much smaller than the SMS database(s) at head office. Those store-level databases normally use the simple recovery model.

An automated job within the SMS software performs nightly full backup and maintenance for those store-level databases, including dbcc checkdb (full), index maintenance and statistics maintenance. If an integrity error on a SMS database is detected by the SMS software (uses dbcc checkdb), then the SMS software will automatically attempt to repair the database using dbcc checkdb with repair_rebuild. If that repair fails (would appear as an error message in the SMS logs), then a support call should be placed with the appropriate provider of database support services. The Messages screen in SMS Pro at store level should be used to check for messages from the automated maintenance at store level.

Backup – verification and testing

Database backup files should periodically be tested in order to ensure that they can be successfully loaded (restored). This would reduce the risk of possibly losing data after discovering that a database  backup file is unusable, e.g. corrupted, once that backup file is needed in the event of a restore scenario. Such testing is usually done by restoring the live backup file on a test RDBMS environment and then running DBCC CheckDB (full) on the test RDBMS environment. As part of the test, copying the live backup file to separate storage on the test environment can be helpful to identify possible file corruption on the source storage device that is used by the live environment. The minimum frequency of testing restores should be once a week for a head office environment.

Verification of database integrity

Database integrity at head office should be verified daily by a scheduled job that executes DBCC CheckDB and reports any errors. If any integrity errors are reported, the database should be repaired in a timely fashion in order to minimize the risk of possible database corruption/data loss. For SMS databases at head office, it is recommended to use DBCC CheckDB with Repair_Rebuild as the first attempt at repair since that does not add any further risk of data loss. For any attempt at repairing database integrities at head office, it is recommended that a full database backup should be done before attempting the repair. If that repair fails, i.e. if the next dbcc checkdb still reports errors for that database, then disaster recovery plans should be followed in order to recover the database from previous database backup files, or possibly from the secondary server in a clustered environment.

In the situation of a serious error such as an error reported by DBCC CheckDB at head office, it is also recommended to backup the current transaction log contents before attempting to fix the problem. This will retain any committed transactions from the log in the event of a restore scenario. If the status of the database is suspect or RECOVERY_PENDING, then it will be necessary to use the NO_TRUNCATE option for the BACKUP command when backing up the transaction log. Backing up the last available transaction log in this manner will allow the option of restoring that transaction log in a full restore / recovery scenario. This procedure is also known as backing up the tail of the transaction log.

Fragmentation – preventative maintenance

The fragmentation level of indices within a database should be monitored by an automated job that also reduces index fragmentation by periodically rebuilding badly fragmented indices (> 30 percent fragmentation) and by re-organizing those indices that have fragmentation levels between 5 and 30 percent. Indices that contain fewer than 1000 index pages should be excluded because de-fragmentation of small tables does not actually improve application performance. This is a standard recommendation from Microsoft. The intent is to maintain the performance of queries that access the database while also minimizing the duration of the job that rebuilds and re-organizes indices. For indices that are re-organized as above, the automated job should also update statistics on the table that is associated to the index.

The automated index maintenance in SMS now performs these operations as of SMS 3.4 . When the automated index maintenance is executed regularly on a maintenance schedule, then indices will usually be reorganized rather than rebuilt.

In order to minimize possible contention with other maintenance jobs and with SMS processing at head office, the online option (only available in SQL Server Enterprise edition) should be used at head office when rebuilding indices. It is also recommended to use SORT_IN_TEMPDB = ON when rebuilding indices in order to minimize space usage in the SMS DB during index rebuilds and also to take advantage of TempDB performance during index rebuilds since TempDB could possibly be configured to use a faster disk subsystem. SQL Server 2014 and above include some improvements to reduce physical i/o in TempDB .

Statistics – preventative maintenance

Database statistics should be updated on a regular basis in order to maintain the performance of queries that access the database. For large SMS databases at head office, it has been observed that when statistics are out of date by more than 3 days, then application performance can become significantly degraded.

For all databases that are used by SMS, the auto create and auto update statistics options at the database level should be used. These settings are on by default when SQL Server is installed. When the auto update statistics setting is on, and when approximately 20 percent of rows in a table plus 500 rows have changed, the SQL engine will automatically update the statistics on that table when a subsequent query references the table. There are some additional scenarios where auto update statistics will cause statistics to be updated and a more detailed explanation of the timing of automatic statistics updating can be found on MSDN.

In addition, the SMS automated maintenance of indices now also updates statistics for tables where an index was re-organized. When the SMS automated maintenance rebuilds a fragmented index, the statistics for the associated table are automatically updated by the SQL engine. The intent of using a mixture of auto updates of statistics as well as a scheduled statistics updating job is to spread the processing overhead of updating statistics across the day and across the week so that noticeable slowdowns of the SQL Server engine can be better managed and prevented.

Database shrink

The Database shrink (DBCC SHRINKDATABASE) should not be used for regular database maintenance in a production environment because it can cause fragmentation, while not providing any benefit to a production environment because the database will usually grow back to its pre-shrink size within a relatively short time. Using DBCC SHRINKDATABASE can also cause performance degradation when SQL Server later requests the re-allocation of Windows file space as the database data and log files grow back to their former sizes.

Scheduling – preventative maintenance

In order to minimize the possible impact on SQL Server performance in a head office environment, it is recommended to schedule the most resource-intensive maintenance jobs, such as backup and DBCC CheckDB to run at a time of day when SMS is likely to be less busy, such as between 6 PM and 11:45 PM. DBCC CheckDB and backup (full or differential) should be stacked or scheduled so that both cannot run at the same time on the same database.

At store level, the SMS software stacks DBCC CheckDB and full backup as part of daily maintenance. While the databases at store level are usually much smaller than the database(s) at head office, it is still recommended to schedule (using SMS) store-level database maintenance at a less busy time of day such as overnight when the store is closed.

Routine database monitoring and maintenance activities should be automated so that they do not require human intervention beyond looking at reported errors on an infrequent basis. This is especially desirable for environments that use many SQL Server instances, in order to maximize the productivity of personnel.

Scheduling – recommendations when using a clustered environment

When using MSSQL database mirroring, for example when using a clustered SQL Server environment at head office or when maintaining a replica using Always On Availability Groups in SQL2014 and above, backup and DBCC CheckDB should ideally be scheduled to run on both the primary and the secondary servers in order to reduce the chance of possible data loss in certain failover scenarios. However, if performance considerations prevent running DBCC CheckDB on both servers, then DBCC CheckDB should at least be run on the server where the daily backup is run and it should be scheduled to run before the backup.

Index maintenance and statistics maintenance jobs should be scheduled to run on the primary server.
In such environments, it is not necessary to also schedule index maintenance jobs or statistics maintenance jobs on the secondary server since the data replication process automatically updates those structures on the secondary server. Please refer to the Always On links (below) for additional recommendations, and for explanations of some failover scenarios.
